### 10 Паттернов для Проектирования Промпт-Модулей

Вот десяток вариантов. Разные инструменты для разных задач.

**1. Паттерн: "Персона" (Persona Sheet)**
*   **Аналогия:** Лист персонажа из D&D или бриф на креативную личность.
*   **Суть:** Максимально глубоко описывает РОЛЬ, которую должна отыграть модель. Не просто "ты аналитик", а целая легенда.
*   **Применение:** Для сложных творческих или экспертных задач, где важен стиль, бэкграунд и система ценностей.
*   **Структура:**
    *   `ROLE`: (Должность)
    *   `BACKSTORY`: (Легенда, откуда взялся этот опыт)
    *   `TRAITS`: (Черты характера: дотошный, креативный, циничный)
    *   `KNOWLEDGE_BASE`: (Области экспертизы)
    *   `QUIRKS`: (Странности или особые подходы)

**2. Паттерн: "Спецификация" (API Spec)**
*   **Аналогия:** Описание функции в коде или эндпоинта в Swagger.
*   **Суть:** Ультимативно сухой, технический, недвусмысленный контракт на генерацию данных. Ноль креатива.
*   **Применение:** Для задач извлечения и структурирования данных. Идеально для `Structured Output`.
*   **Структура:**
    *   `FUNCTION_NAME`: (Название операции)
    *   `DESCRIPTION`: (Что она делает)
    *   `INPUT_PARAMS`: (Описание входных данных и их формат)
    *   `RETURN_SCHEMA`: (JSON Schema или описание формата вывода)
    *   `ERROR_CODES`: (Как реагировать на невалидные данные)

**3. Паттерн: "Ограничитель" (Guardrail)**
*   **Аналогия:** Брандмауэр, система линтинга кода, цензурный комитет.
*   **Суть:** Модуль, который не говорит, *что* делать, а говорит, *чего НЕ делать*. Сосредоточен исключительно на ограничениях.
*   **Применение:** Для задач, требующих высокой безопасности, этичности или соответствия строгим правилам. Используется как "обертка" поверх других промптов.
*   **Структура:**
    *   `FORBIDDEN_TOPICS`: (Запрещенные темы)
    *   `CONTENT_CONSTRAINTS`: (Не генерировать PII, код с уязвимостями, и т.д.)
    *   `STYLE_VIOLATIONS`: (Не использовать ненормативную лексику, пассивно-агрессивный тон...)
    *   `FAILURE_POLICY`: (Что делать при невозможности выполнить правила: "Ответить отказом", "Запросить уточнение")

**4. Паттерн: "Стильгайд" (Style Guide)**
*   **Аналогия:** Брендбук, редакционная политика.
*   **Суть:** Фокусируется на стиле, тоне и голосе (Tone of Voice).
*   **Применение:** Для всех задач генерации человекочитаемого текста (копирайтинг, ответы поддержки).
*   **Структура:**
    *   `TONE`: (Формальный, Дружелюбный, Остроумный, Эмпатичный)
    *   `VOICE`: (От первого лица, от третьего, от имени бренда "Мы")
    *   `LEXICON`: (Предпочтительная лексика: "клиент" vs "пользователь")
    *   `DOs_AND_DONTs`: (Примеры хорошего и плохого стиля)

**5. Паттерн: "Аксиома" (Axiomatic Principle)**
*   **Аналогия:** Первый закон робототехники, философский принцип.
*   **Суть:** Формулирует один-единственный, фундаментальный, непреложный принцип, который должен лежать в основе всех рассуждений модели.
*   **Применение:** Как глобальный `system_instruction` для всей системы или для агентов, чье поведение должно быть подчинено одной главной идее.
*   **Структура:**
    *   `CORE_PRINCIPLE`: ("Всегда ставь ясность и простоту выше технической сложности.")
    *   `IMPLICATIONS`: ("Это означает: избегай жаргона, используй короткие предложения, предлагай самый очевидный путь решения.")

**6. Паттерн: "Критик" (Critique Framework)**
*   **Аналогия:** Форма для рецензирования научной статьи или код-ревью.
*   **Суть:** Предоставляет модели структуру для оценки и критики предоставленных данных. Не говорит, *как* критиковать, а говорит, *по каким критериям*.
*   **Применение:** Для `CRITIQUE` операций в DCL, для этапов рефлексии.
*   **Структура:**
    *   `INPUT_ARTIFACT`: (Что мы оцениваем: `[ТЕКСТ]`, `[СХЕМА]`)
    *   `EVALUATION_CRITERIA`: (Список критериев: `[Полнота, Логическая консистентность, Соответствие требованиям, Ясность]`)
    *   `OUTPUT_FORMAT`: (Как должна выглядеть критика: "Список достоинств", "Список недостатков", "Рекомендации по улучшению")

**7. Паттерн: "Экспонат" (Exemplar / Few-shot)**
*   **Аналогия:** Зал с образцами в музее, папка `examples` в репозитории.
*   **Суть:** Модуль, который не содержит инструкций, а только высококачественные примеры пар `[Вход] -> [Выход]`.
*   **Применение:** Для сложных задач трансформации формата или стиля, где "показать" работает лучше, чем "рассказать".
*   **Структура:**
    *   `EXAMPLE_1`: { `input`: "...", `output`: "..." }
    *   `EXAMPLE_2`: { `input`: "...", `output`: "..." }
    *   ...

**8. Паттерн: "Польза" (Goal-Benefit / User Story)**
*   **Аналогия:** User Story в Agile.
*   **Суть:** Формулирует задачу через конечную пользу для пользователя.
*   **Применение:** Для агентов, которые должны генерировать решения или идеи, ориентированные на пользователя.
*   **Структура:**
    *   `AS_A`: (Роль конечного пользователя: "Как продавец на C2C-маркетплейсе...")
    *   `I_WANT_TO`: (Цель: "...получить описание товара, которое...")
    *   `SO_THAT`: (Польза: "...я мог продать его быстрее и дороже.")

**9. Паттерн: "Источник Данных" (Data Source)**
*   **Аналогия:** `Context` в React, `dataProvider` в других фреймворках.
*   **Суть:** Модуль, предназначенный исключительно для структурированной подачи "сырых" данных в LLM.
*   **Применение:** Когда нужно "заземлить" модель на большом объеме фактической информации для анализа или RAG.
*   **Структура:**
    *   `DATA_TYPE`: (Тип данных: `CSV`, `JSON`, `Текстовые выдержки`)
    *   `SCHEMA`: (Описание структуры данных)
    *   `PAYLOAD`: (`{{actual_data}}`)

**10. Паттерн: "Чек-лист" (Checklist)**
*   **Аналогия:** Предполетный чек-лист пилота.
*   **Суть:** Предоставляет список бинарных (да/нет) или категориальных утверждений, которые модель должна проверить или заполнить на основе входных данных.
*   **Применение:** Для задач валидации, комплаенса, простой классификации.
*   **Структура:**
    *   `INPUT_DOCUMENT`: `{{text_to_analyze}}`
    *   `CHECKLIST_ITEMS`:

### 10 Паттернов Проектирования `Prompt Modules`

#### 1. Паттерн: "Императивный Манифест" (`R.I.C.E.`)
*   **Философия:** Прямая, как палка, инструкция. "Делай раз, делай два, делай три". Идеален для простых, детерминированных задач.
*   **Компоненты:**
    *   **R**ole (Роль): "Ты — переводчик с русского на английский."
    *   **I**nstruction (Инструкция): "Переведи следующий текст."
    *   **C**onstraints (Ограничения): "Стиль — формальный. Не используй идиомы."
    *   **E**xemplar (Пример): "Вход: 'Привет, мир!' -> Выход: 'Hello, world!'"
*   **Применение:** Переводы, форматирование текста, простые извлечения данных.

#### 2. Паттерн: "Декларативная Цель" (`G.O.A.L.`)
*   **Философия:** Описываем конечный результат и критерии его приемки, но не говорим, как его достичь. Даем свободу модели.
*   **Компоненты:**
    *   **G**oal (Цель): "Создать три уникальных рекламных слогана для кофейни."
    *   **O**bjectives (Задачи/Критерии): "Слоганы должны быть короткими (3-5 слов), вызывать чувство уюта, упоминать speciality-кофе."
    *   **A**udience (Аудитория): "Целевая аудитория — фрилансеры и студенты 20-30 лет."
    *   **L**imits (Границы): "Не использовать слова: 'лучший', 'энергия', 'бодрость'."
*   **Применение:** Креативная работа, генерация идей, написание маркетинговых текстов.

#### 3. Паттерн: "Контракт на Оказание Услуг" (`S.L.A.`)
*   **Философия:** Максимально формализованный подход для задач, где цена ошибки высока. Промпт строится как юридический документ.
*   **Компоненты:**
    *   **S**cope (Объем работ): "Проанализировать отзыв клиента на предмет тональности (позитив/негатив/нейтрал) и извлечь упомянутые продукты."
    *   **L**iabilities (Ограничение ответственности): "ЗАПРЕЩЕНО: Делать выводы о личности клиента. Интерпретировать сарказм. Если тональность неясна, вернуть 'ambiguous'."
    *   **A**cceptance Criteria (Критерии приемки): "Результат должен быть JSON-объектом вида `{'sentiment': '...', 'products': [...]}`. Поле `products` должно быть пустым массивом, если продукты не упомянуты."
*   **Применение:** Анализ юридических документов, модерация контента, задачи с высокими требованиями к безопасности и точности.

#### 4. Паттерн: "Цепное рассуждение" (Chain of Thought - `CoT`)
*   **Философия:** Заставляем модель "думать вслух" перед тем, как дать финальный ответ. Классика для сложных логических задач.
*   **Компоненты:**
    *   **Задача:** "Сколько теннисных мячей поместится в школьный автобус?"
    *   **Мета-инструкция:** "Прежде чем дать ответ, пошагово распиши всю цепочку твоих рассуждений. Оцени размеры автобуса, размеры мяча, коэффициент упаковки и т.д."
    *   **Формат вывода:** "Сначала секция 'Рассуждения:', затем секция 'Ответ:'."
*   **Применение:** Математические задачи, логические головоломки, планирование, оценка сложных систем.

#### 5. Паттерн: "Команда Экспертов" (Panel of Experts)
*   **Философия:** Симуляция обсуждения проблемы несколькими разными специалистами для получения всестороннего ответа.
*   **Компоненты:**
    *   **Задача:** "Оценить бизнес-идею 'подписка на носки'."
    *   **Роли:** "Представь, что ты — команда из трех экспертов:\n1. **Маркетолог:** Оценит рынок и аудиторию.\n2. **Финансист:** Рассчитает юнит-экономику.\n3. **Скептик:** Найдет все слабые места и риски."
    *   **Формат вывода:** "Представь ответ в виде диалога или отчета от каждого эксперта."
*   **Применение:** Стратегический анализ, оценка рисков, генерация разносторонних идей.

#### 6. Паттерн: "Тест-ориентированная разработка" (Test-Driven Development - `TDD`)
*   **Философия:** Даем модели набор тестов, которые ее результат должен пройти. Идеально для кода и структурированных данных.
*   **Компоненты:**
    *   **Задача:** "Напиши функцию `is_palindrome(s)` на Python."

*   **Применение:** Генерация кода, написание SQL-запросов, создание RegEx-выражений, генерация структурированных данных.

#### 7. Паттерн: "Перевертыш" (Flipped Interaction)
*   **Философия:** Модель не отвечает на вопрос, а задает их, чтобы помочь пользователю самому прийти к решению.
*   **Компоненты:**
    *   **Роль:** "Ты — Сократ, философ. Твоя задача — не давать прямых ответов, а с помощью наводящих вопросов помочь мне разобраться в проблеме."
    *   **Проблема:** "Я не могу выбрать между карьерой программиста и дизайнера."
    *   **Инструкция:** "Твой первый вопрос должен помочь мне определить мои ключевые ценности."
*   **Применение:** Коучинг, самоанализ, образовательные задачи, решение плохо формализованных проблем.

#### 8. Паттерн: "Аналогия" (Analogy Mapping)
*   **Философия:** Объяснить сложную концепцию, проведя параллели с простой и понятной.
*   **Компоненты:**
    *   **Задача:** "Объясни, что такое Kubernetes."
    *   **Аналогия:** "Объясни это на аналогии с дирижером и оркестром. Оркестр — это сервера, музыканты — контейнеры, а Kubernetes — это дирижер, который следит, чтобы все играли слаженно."
*   **Применение:** Объяснение сложных тем, создание обучающих материалов, онбординг.

#### 9. Паттерн: "Строитель из примитивов" (Primitive Builder)
*   **Философия:** Модель не генерирует конечный результат, а выдает набор "строительных блоков", из которых его можно собрать.
*   **Компоненты:**
    *   **Задача:** "Придумай концепцию мобильной игры."
    *   **Примитивы:** "Твой ответ должен состоять из следующих блоков:\n- **Core Loop:** (Основной игровой цикл)\n- **Monetization:** (Модель монетизации)\n- **Unique Selling Point:** (Уникальное торговое предложение)"
*   **Применение:** Декомпозиция сложных задач, гейм-дизайн, продуктовая разработка.

#### 10. Паттерн: "Рефлексивный критик" (Reflective Critic)
*   **Философия:** Двухэтапный промпт. Сначала модель генерирует ответ, потом сама же его критикует и улучшает. Полностью соответствует идеям Andrew Ng.
*   **Компоненты:**
    *   **Шаг 1 (Генератор):** "Напиши короткое эссе о влиянии соцсетей на общество."
    *   **Шаг 2 (Критик - может быть в том же промпте):** "А теперь выступи в роли предвзятого критика. Найди в написанном эссе три слабых аргумента и предложи, как их усилить."
*   **Применение:** Улучшение качества текста, кода, планов. Повышение надежности и глубины ответов.

### **Группа 1: Формальные / "Контрактные" Фреймворки**

(Подход: промпт как строгий технический или юридический документ)

1.  **The Legal Brief (Юридическая Записка):** Структура `[Факты] -> [Применимые Законы/Правила] -> [Анализ] -> [Заключение]`. Идеально для задач, требующих строгого следования правилам.

2.  **The API Contract (Контракт API):** Структура `[Endpoint/Function Name] -> [Description] -> [Parameters (Name, Type, Required)] -> [Return Value Schema]`. Для задач, где LLM должен сгенерировать что-то строго структурированное.

3.  **The Scientific Paper (IMRaD):** Структура `[Introduction/Goal] -> [Methods/Process] -> [Results/Output Spec] -> [Discussion/Reflection]`. Отлично подходит для исследовательских и аналитических задач.

### **Группа 2: Ролевые / "Персонажные" Фреймворки**

(Подход: промпт как сценарий или инструкция для актера)

4.  **The Actor's Method (Система Станиславского):** Структура `[Роль] -> [Сверхзадача] -> [Предлагаемые Обстоятельства (Контекст)] -> [Действие]`. Глубокое погружение модели в роль для генерации очень стилизованных ответов.

5.  **The Expert Panel (Экспертный Совет):** Моделируется диалог нескольких ролей. `[Модератор: Цель] -> [Эксперт A: Взгляд с одной стороны] -> [Эксперт B: Взгляд с другой стороны] -> [Аналитик: Синтез и вывод]`. Для генерации сбалансированных, многогранных ответов.

6.  **The Press Conference (Пресс-конференция):** Структура `[Главный Спикер: Основной тезис] -> [Журналисты: Набор острых вопросов, на которые нужно ответить] -> [Спикер: Финальное заявление]`. Заставляет модель предвидеть и отвечать на потенциальную критику.

### **Группа 3: Процессные / "Алгоритмические" Фреймворки**

(Подход: промпт как алгоритм или описание процесса)

7.  **The Pseudocode Spec (Спецификация в Псевдокоде):** `FUNCTION SolveProblem(input):\n // Step 1: Analyze input...\n // Step 2: Generate draft...\n // Step 3: Verify constraints...\n RETURN result`. Для задач, требующих строгой последовательности логических шагов.

8.  **The User Story (BDD/Gherkin):** `FEATURE: [Название фичи]\n SCENARIO: [Название сценария]\n GIVEN [Исходный контекст]\n WHEN [Происходит действие]\n THEN [Ожидаемый результат]`. Классика для описания поведения.

9.  **The Assembly Line (Сборочный Конвейер):** `[Станция 1: Прием Сырья (Input)] -> [Станция 2: Первичная Обработка (Generate)] -> [Станция 3: Контроль Качества (Critique)] -> [Станция 4: Упаковка (Format Output)]`. Линейный, пошаговый процесс.

### **Группа 4: Минималистичные / "Ограничительные" Фреймворки**

(Подход: минимум инструкций, максимум свободы внутри жестких рамок)

10. **The Zen Koan (Дзенский Коан):** Только `[Цель]` и один `[Ключевой парадокс/вопрос]`. Дает модели максимальную свободу для "творческого" решения внутри очень узкого концептуального коридора.

11. **The Guardrail Protocol (Протокол Ограждений):** Только `[Цель]` и `[Запреты (DO NOT...)]`. Вместо того чтобы говорить, что делать, мы говорим, чего НЕ делать. Полезно для избегания типичных ошибок.

12. **The Sandbox (Песочница):** Только `[Начальное состояние]` и `[Разрешенные действия/инструменты]`. Модель сама решает, как комбинировать разрешенные действия для достижения неявной цели.

### **Группа 5: Данно-центричные / "Примерные" Фреймворки**

(Подход: данные и примеры важнее инструкций)

13. **The Data Frame (Таблица Данных):** Промпт состоит преимущественно из структурированных данных (Markdown таблица, CSV), с минимальной инструкцией "Проанализируй и сделай вывод".

14. **The Example-Driven Template (Шаблон на Примерах):** Классический few-shot, но формализованный. `TEMPLATE:\n Input: {{input}}\n Output: {{output}}\n EXAMPLES:\n [Пример 1]\n [Пример 2]\n EXECUTE:\n Input: [Новый Вход]\n Output:`.

15. **The Contrastive Pair (Контрастная Пара):** `[Цель]\n [Хороший Пример (Good Example)]\n [Плохой Пример (Bad Example) и почему он плох]\n [Твоя задача]`. Обучение на анти-примерах, один из самых мощных методов.

### 15+ Фреймворков для Проектирования `Prompt Modules`

#### Группа 1: Аналогии из Инженерных и Гуманитарных Дисциплин

Эти фреймворки заимствуют проверенные временем структуры из других областей.

1.  **The User Story (`AS A / I WANT / SO THAT`)**
    *   **Структура:** `AS A [Роль/Персона], I WANT [Действие/Цель], SO THAT [Бизнес-ценность]`.
    *   **Применение:** Идеально для постановки бизнес-ориентированных задач. Заставляет сразу думать о конечной ценности.

2.  **The Gherkin (`GIVEN / WHEN / THEN`)**
    *   **Структура:** `GIVEN [Исходное состояние/Контекст], WHEN [Событие/Действие], THEN [Ожидаемый результат]`.
    *   **Применение:** Для задач, где важна логика "если-то". Отлично подходит для написания правил валидации, классификации или триггеров.

3.  **The Legal Brief (`IRAC`)**
    *   **Структура:** `ISSUE [Проблема], RULE [Применимое правило/Принцип], APPLICATION [Применение правила к фактам], CONCLUSION [Вывод]`.
    *   **Применение:** Для сложных аналитических задач, требующих логического вывода на основе набора правил (например, наш фреймворк "Линз").

4.  **The 5 Ws (`WHO / WHAT / WHEN / WHERE / WHY`)**
    *   **Структура:** Пять классических журналистских вопросов для сбора полной картины.
    *   **Применение:** Отлично подходит для генерации саммари, отчетов или первоначального сбора требований по задаче.

5.  **The Scientific Method (`HYPOTHESIS / EXPERIMENT / RESULTS / CONCLUSION`)**
    *   **Структура:** Формализация исследовательского процесса.
    *   **Применение:** Для агентов, которые должны анализировать данные, находить паттерны и делать выводы.

#### Группа 2: Структурно-Компонентные Фреймворки

Эти фреймворки фокусируются на явном определении всех необходимых компонентов для выполнения задачи.

6.  **The R.O.P.E.S.**
    *   **Структура:** `ROLE / OBJECTIVE / PROCESS / EXEMPLARS / SPECIFICATION (Output)`.
    *   **Применение:** Очень полный и строгий фреймворк, похожий на то, что мы обсуждали. Хорош для сложных, критически важных задач.

7.  **The C.O.S.T.A.R.**
    *   **Структура:** `CONTEXT / OBJECTIVE / STYLE / TONE / AUDIENCE / RESPONSE (Format)`.
    *   **Применение:** Сильный акцент на "мягких" аспектах генерации контента. Идеален для Storyteller-агентов.

8.  **The S.P.E.C.**
    *   **Структура:** `SITUATION [Описание текущего положения], PURPOSE [Цель операции], EXPECTATIONS [Детальные ожидания от результата], CRITERIA [Критерии оценки успеха]`.
    *   **Применение:** Хорош для задач, где важна последующая автоматическая или ручная оценка качества результата.

9.  **The I.P.O. (`INPUT / PROCESSING / OUTPUT`)**
    *   **Структура:** Классическая модель из информатики. `INPUT [Входные данные], PROCESSING [Шаги обработки], OUTPUT [Спецификация вывода]`.
    *   **Применение:** Универсальный и простой фреймворк для большинства задач по трансформации данных.

#### Группа 3: Целе-ориентированные и Копирайтерские Фреймворки

Эти фреймворки заточены под конкретный тип результата: убеждение, создание, анализ.

10. **The P.A.S. (`PROBLEM / AGITATE / SOLVE`)**
    *   **Структура:** `PROBLEM [Опиши проблему аудитории], AGITATE [Усугуби эту проблему, покажи ее последствия], SOLVE [Предложи решение]`.
    *   **Применение:** Классика копирайтинга. Идеально для генерации продающих текстов, маркетинговых материалов.

11. **The A.I.D.A. (`ATTENTION / INTEREST / DESIRE / ACTION`)**
    *   **Структура:** Еще одна классическая маркетинговая воронка.
    *   **Применение:** Для генерации контента, который должен провести пользователя по определенному пути.

12. **The Hegelian Dialectic (`THESIS / ANTITHESIS / SYNTHESIS`)**
    *   **Структура:** `THESIS [Изложи исходное утверждение], ANTITHESIS [Сформулируй противоположное утверждение], SYNTHESIS [Создай новый вывод, объединяющий или разрешающий противоречие]`.
    *   **Применение:** Для глубокого анализа сложных тем, генерации разносторонних аргументов или создания сбалансированного обзора.

#### Группа 4: Минималистичные и Специализированные Фреймворки

Для простых или очень специфичных задач.

13. **The Command (`VERB / NOUN / CONSTRAINTS`)**
    *   **Структура:** `[Действие] [Объект] [Ограничения]`.
    *   **Применение:** Для простых, унитарных задач. `GENERATE list of 10 ideas FOR blog_post ABOUT 'AI agents'`.

14. **The Transformer (`INPUT / FORMAT_FROM / FORMAT_TO`)**
    *   **Структура:** `INPUT [Данные], FORMAT_FROM [Исходный формат], FORMAT_TO [Целевой формат]`.
    *   **Применение:** Специализированный фреймворк для задач по преобразованию форматов (JSON в YAML, Markdown в HTML и т.д.).

15. **The Reflector (`INPUT / CRITERIA / FEEDBACK`)**
    *   **Структура:** `INPUT [Черновик для анализа], CRITERIA [Список критериев для оценки], FEEDBACK [Формат для вывода критики]`.
    *   **Применение:** Специализированный фреймворк для реализации паттерна "Рефлексия", о котором говорил Andrew Ng.

Хороший заход. Если отбросить шелуху, то принципиальная разница в том, кто является **исполнителем** (актором).

В UC, User Story и UML мы описываем деятельность для **человека** (разработчика, аналитика, пользователя). Мы полагаемся на его когнитивные способности, фоновые знания и умение интерпретировать нечеткости. "Система должна позволять администратору..." — человек-разработчик поймет, о чем речь.

В `Prompt Module` исполнителем является **LLM**. Это не человек. У нее нет фоновых знаний о *вашем* проекте (кроме тех, что в промпте), но есть гигантские фоновые знания о *мире*. У нее нет интуиции, но есть вероятностное предсказание.

**Принципиальные отличия `Prompt Module` от UC/User Story:**

1.  **Инструктивность vs Описательность (Imperative vs Descriptive):**
    *   User Story: "Я как пользователь хочу..." (Описывает *желание*).
    *   Prompt Module: "Ты — эксперт. Твоя задача — сделать X..." (Дает прямую *команду* к действию).
    *   Промпт — это **исполняемый код на естественном языке**. Это ближе к функции в программировании, чем к требованию в Jira.

2.  **Контекстная Изоляция (Self-Contained Context):**
    *   User Story существует в контексте бэклога и знаний команды.
    *   Prompt Module должен быть самодостаточным. LLM ничего не знает о том, что было "на митинге". Все, что нужно для выполнения, должно быть внутри модуля или передано явно. Отсюда необходимость в жестком разделе `Input Data / Facts`.

3.  **Примеры как Спецификация (Examples as Spec):**
    *   В UML мы рисуем диаграмму классов.
    *   В Промпте мы даем few-shot примеры. Для LLM примеры работают лучше, чем абстрактные правила. В требованиях к ПО примеры — это "пояснение", в промптах — это "обучающая выборка". Это фундаментально иной способ передачи знания.

4.  **Управление "Личностью" (Persona/Role Engineering):**
    *   В требованиях нам все равно, в каком "настроении" код выполняет функцию.
    *   В Промпте задание Роли ("Ты — скептичный критик") меняет *качество* и *характер* результата. Это рычаг управления, которого нет в классическом проектировании ПО.

**Предложение по структуре (Исправленное):**

Если мы не можем использовать слова `Context` и `Process` (потому что они заняты на уровнях выше), давай использовать термины, более близкие к программированию функций или написанию сценариев:

1.  **`Instruction / Directive` (Вместо Process):** Основная команда. Что именно делать. Алгоритм действий.
2.  **`Inputs / Data` (Вместо Context):** Плейсхолдеры для входных данных. Аргументы функции.
3.  **`Persona / Stance`:** Настройка "движка".
4.  **`Constraints / Rules`:** Инварианты и ограничения.
5.  **`Output Format`:** Тип возвращаемого значения.

Такая структура подчеркивает, что `Prompt Module` — это функциональный блок, а не просто описание пожеланий.


**1. Паттерн: "Персона" (Persona Sheet)**
*   **Аналогия:** Лист персонажа из D&D или бриф на креативную личность.
*   **Суть:** Максимально глубоко описывает РОЛЬ, которую должна отыграть модель. Не просто "ты аналитик", а целая легенда.
*   **Применение:** Для сложных творческих или экспертных задач, где важен стиль, бэкграунд и система ценностей.
*   **Структура:**
    *   `ROLE`: (Должность)
    *   `BACKSTORY`: (Легенда, откуда взялся этот опыт)
    *   `TRAITS`: (Черты характера: дотошный, креативный, циничный)
    *   `KNOWLEDGE_BASE`: (Области экспертизы)
    *   `QUIRKS`: (Странности или особые подходы)

**2. Паттерн: "Спецификация" (API Spec)**
*   **Аналогия:** Описание функции в коде или эндпоинта в Swagger.
*   **Суть:** Ультимативно сухой, технический, недвусмысленный контракт на генерацию данных. Ноль креатива.
*   **Применение:** Для задач извлечения и структурирования данных. Идеально для `Structured Output`.
*   **Структура:**
    *   `FUNCTION_NAME`: (Название операции)
    *   `DESCRIPTION`: (Что она делает)
    *   `INPUT_PARAMS`: (Описание входных данных и их формат)
    *   `RETURN_SCHEMA`: (JSON Schema или описание формата вывода)
    *   `ERROR_CODES`: (Как реагировать на невалидные данные)

**3. Паттерн: "Ограничитель" (Guardrail)**
*   **Аналогия:** Брандмауэр, система линтинга кода, цензурный комитет.
*   **Суть:** Модуль, который не говорит, *что* делать, а говорит, *чего НЕ делать*. Сосредоточен исключительно на ограничениях.
*   **Применение:** Для задач, требующих высокой безопасности, этичности или соответствия строгим правилам. Используется как "обертка" поверх других промптов.
*   **Структура:**
    *   `FORBIDDEN_TOPICS`: (Запрещенные темы)
    *   `CONTENT_CONSTRAINTS`: (Не генерировать PII, код с уязвимостями, и т.д.)
    *   `STYLE_VIOLATIONS`: (Не использовать ненормативную лексику, пассивно-агрессивный тон...)
    *   `FAILURE_POLICY`: (Что делать при невозможности выполнить правила: "Ответить отказом", "Запросить уточнение")

**4. Паттерн: "Стильгайд" (Style Guide)**
*   **Аналогия:** Брендбук, редакционная политика.
*   **Суть:** Фокусируется на стиле, тоне и голосе (Tone of Voice).
*   **Применение:** Для всех задач генерации человекочитаемого текста (копирайтинг, ответы поддержки).
*   **Структура:**
    *   `TONE`: (Формальный, Дружелюбный, Остроумный, Эмпатичный)
    *   `VOICE`: (От первого лица, от третьего, от имени бренда "Мы")
    *   `LEXICON`: (Предпочтительная лексика: "клиент" vs "пользователь")
    *   `DOs_AND_DONTs`: (Примеры хорошего и плохого стиля)

**5. Паттерн: "Аксиома" (Axiomatic Principle)**
*   **Аналогия:** Первый закон робототехники, философский принцип.
*   **Суть:** Формулирует один-единственный, фундаментальный, непреложный принцип, который должен лежать в основе всех рассуждений модели.
*   **Применение:** Как глобальный `system_instruction` для всей системы или для агентов, чье поведение должно быть подчинено одной главной идее.
*   **Структура:**
    *   `CORE_PRINCIPLE`: ("Всегда ставь ясность и простоту выше технической сложности.")
    *   `IMPLICATIONS`: ("Это означает: избегай жаргона, используй короткие предложения, предлагай самый очевидный путь решения.")

**6. Паттерн: "Критик" (Critique Framework)**
*   **Аналогия:** Форма для рецензирования научной статьи или код-ревью.
*   **Суть:** Предоставляет модели структуру для оценки и критики предоставленных данных. Не говорит, *как* критиковать, а говорит, *по каким критериям*.
*   **Применение:** Для `CRITIQUE` операций в DCL, для этапов рефлексии.
*   **Структура:**
    *   `INPUT_ARTIFACT`: (Что мы оцениваем: `[ТЕКСТ]`, `[СХЕМА]`)
    *   `EVALUATION_CRITERIA`: (Список критериев: `[Полнота, Логическая консистентность, Соответствие требованиям, Ясность]`)
    *   `OUTPUT_FORMAT`: (Как должна выглядеть критика: "Список достоинств", "Список недостатков", "Рекомендации по улучшению")

**7. Паттерн: "Экспонат" (Exemplar / Few-shot)**
*   **Аналогия:** Зал с образцами в музее, папка `examples` в репозитории.
*   **Суть:** Модуль, который не содержит инструкций, а только высококачественные примеры пар `[Вход] -> [Выход]`.
*   **Применение:** Для сложных задач трансформации формата или стиля, где "показать" работает лучше, чем "рассказать".
*   **Структура:**
    *   `EXAMPLE_1`: { `input`: "...", `output`: "..." }
    *   `EXAMPLE_2`: { `input`: "...", `output`: "..." }
    *   ...

**8. Паттерн: "Польза" (Goal-Benefit / User Story)**
*   **Аналогия:** User Story в Agile.
*   **Суть:** Формулирует задачу через конечную пользу для пользователя.
*   **Применение:** Для агентов, которые должны генерировать решения или идеи, ориентированные на пользователя.
*   **Структура:**
    *   `AS_A`: (Роль конечного пользователя: "Как продавец на C2C-маркетплейсе...")
    *   `I_WANT_TO`: (Цель: "...получить описание товара, которое...")
    *   `SO_THAT`: (Польза: "...я мог продать его быстрее и дороже.")

**9. Паттерн: "Источник Данных" (Data Source)**
*   **Аналогия:** `Context` в React, `dataProvider` в других фреймворках.
*   **Суть:** Модуль, предназначенный исключительно для структурированной подачи "сырых" данных в LLM.
*   **Применение:** Когда нужно "заземлить" модель на большом объеме фактической информации для анализа или RAG.
*   **Структура:**
    *   `DATA_TYPE`: (Тип данных: `CSV`, `JSON`, `Текстовые выдержки`)
    *   `SCHEMA`: (Описание структуры данных)
    *   `PAYLOAD`: (`{{actual_data}}`)

**10. Паттерн: "Чек-лист" (Checklist)**
*   **Аналогия:** Предполетный чек-лист пилота.
*   **Суть:** Предоставляет список бинарных (да/нет) или категориальных утверждений, которые модель должна проверить или заполнить на основе входных данных.
*   **Применение:** Для задач валидации, комплаенса, простой классификации.
*   **Структура:**
    *   `INPUT_DOCUMENT`: `{{text_to_analyze}}`
    *   `CHECKLIST_ITEMS`:
        *   `Item_1`: ("Содержит ли текст персональные данные? [ДА/НЕТ]")
        *   `Item_2`: ("Тональность текста: [ПОЗИТИВНАЯ/НЕЙТРАЛЬНАЯ/НЕГАТИВНАЯ]")
        *   `Item_3`: ("Соответствует ли требованиям X? [ДА/НЕТ/ЧАСТИЧНО]")
		
#### 1. Паттерн: "Императивный Манифест" (`R.I.C.E.`)
*   **Философия:** Прямая, как палка, инструкция. "Делай раз, делай два, делай три". Идеален для простых, детерминированных задач.
*   **Компоненты:**
    *   **R**ole (Роль): "Ты — переводчик с русского на английский."
    *   **I**nstruction (Инструкция): "Переведи следующий текст."
    *   **C**onstraints (Ограничения): "Стиль — формальный. Не используй идиомы."
    *   **E**xemplar (Пример): "Вход: 'Привет, мир!' -> Выход: 'Hello, world!'"
*   **Применение:** Переводы, форматирование текста, простые извлечения данных.

#### 2. Паттерн: "Декларативная Цель" (`G.O.A.L.`)
*   **Философия:** Описываем конечный результат и критерии его приемки, но не говорим, как его достичь. Даем свободу модели.
*   **Компоненты:**
    *   **G**oal (Цель): "Создать три уникальных рекламных слогана для кофейни."
    *   **O**bjectives (Задачи/Критерии): "Слоганы должны быть короткими (3-5 слов), вызывать чувство уюта, упоминать speciality-кофе."
    *   **A**udience (Аудитория): "Целевая аудитория — фрилансеры и студенты 20-30 лет."
    *   **L**imits (Границы): "Не использовать слова: 'лучший', 'энергия', 'бодрость'."
*   **Применение:** Креативная работа, генерация идей, написание маркетинговых текстов.

#### 3. Паттерн: "Контракт на Оказание Услуг" (`S.L.A.`)
*   **Философия:** Максимально формализованный подход для задач, где цена ошибки высока. Промпт строится как юридический документ.
*   **Компоненты:**
    *   **S**cope (Объем работ): "Проанализировать отзыв клиента на предмет тональности (позитив/негатив/нейтрал) и извлечь упомянутые продукты."
    *   **L**iabilities (Ограничение ответственности): "ЗАПРЕЩЕНО: Делать выводы о личности клиента. Интерпретировать сарказм. Если тональность неясна, вернуть 'ambiguous'."
    *   **A**cceptance Criteria (Критерии приемки): "Результат должен быть JSON-объектом вида `{'sentiment': '...', 'products': [...]}`. Поле `products` должно быть пустым массивом, если продукты не упомянуты."
*   **Применение:** Анализ юридических документов, модерация контента, задачи с высокими требованиями к безопасности и точности.

#### 4. Паттерн: "Цепное рассуждение" (Chain of Thought - `CoT`)
*   **Философия:** Заставляем модель "думать вслух" перед тем, как дать финальный ответ. Классика для сложных логических задач.
*   **Компоненты:**
    *   **Задача:** "Сколько теннисных мячей поместится в школьный автобус?"
    *   **Мета-инструкция:** "Прежде чем дать ответ, пошагово распиши всю цепочку твоих рассуждений. Оцени размеры автобуса, размеры мяча, коэффициент упаковки и т.д."
    *   **Формат вывода:** "Сначала секция 'Рассуждения:', затем секция 'Ответ:'."
*   **Применение:** Математические задачи, логические головоломки, планирование, оценка сложных систем.

#### 5. Паттерн: "Команда Экспертов" (Panel of Experts)
*   **Философия:** Симуляция обсуждения проблемы несколькими разными специалистами для получения всестороннего ответа.
*   **Компоненты:**
    *   **Задача:** "Оценить бизнес-идею 'подписка на носки'."
    *   **Роли:** "Представь, что ты — команда из трех экспертов:\n1. **Маркетолог:** Оценит рынок и аудиторию.\n2. **Финансист:** Рассчитает юнит-экономику.\n3. **Скептик:** Найдет все слабые места и риски."
    *   **Формат вывода:** "Представь ответ в виде диалога или отчета от каждого эксперта."
*   **Применение:** Стратегический анализ, оценка рисков, генерация разносторонних идей.

#### 6. Паттерн: "Тест-ориентированная разработка" (Test-Driven Development - `TDD`)
*   **Философия:** Даем модели набор тестов, которые ее результат должен пройти. Идеально для кода и структурированных данных.
*   **Компоненты:**
    *   **Задача:** "Напиши функцию `is_palindrome(s)` на Python."
    *   **Тесты (Assertions):** "Твой код должен успешно проходить следующие тесты:\n`assert is_palindrome('madam') == True`\n`assert is_palindrome('step on no pets') == True`\n`assert is_palindrome('hello') == False`"
*   **Применение:** Генерация кода, написание SQL-запросов, создание RegEx-выражений, генерация структурированных данных.

#### 7. Паттерн: "Перевертыш" (Flipped Interaction)
*   **Философия:** Модель не отвечает на вопрос, а задает их, чтобы помочь пользователю самому прийти к решению.
*   **Компоненты:**
    *   **Роль:** "Ты — Сократ, философ. Твоя задача — не давать прямых ответов, а с помощью наводящих вопросов помочь мне разобраться в проблеме."
    *   **Проблема:** "Я не могу выбрать между карьерой программиста и дизайнера."
    *   **Инструкция:** "Твой первый вопрос должен помочь мне определить мои ключевые ценности."
*   **Применение:** Коучинг, самоанализ, образовательные задачи, решение плохо формализованных проблем.

#### 8. Паттерн: "Аналогия" (Analogy Mapping)
*   **Философия:** Объяснить сложную концепцию, проведя параллели с простой и понятной.
*   **Компоненты:**
    *   **Задача:** "Объясни, что такое Kubernetes."
    *   **Аналогия:** "Объясни это на аналогии с дирижером и оркестром. Оркестр — это сервера, музыканты — контейнеры, а Kubernetes — это дирижер, который следит, чтобы все играли слаженно."
*   **Применение:** Объяснение сложных тем, создание обучающих материалов, онбординг.

#### 9. Паттерн: "Строитель из примитивов" (Primitive Builder)
*   **Философия:** Модель не генерирует конечный результат, а выдает набор "строительных блоков", из которых его можно собрать.
*   **Компоненты:**
    *   **Задача:** "Придумай концепцию мобильной игры."
    *   **Примитивы:** "Твой ответ должен состоять из следующих блоков:\n- **Core Loop:** (Основной игровой цикл)\n- **Monetization:** (Модель монетизации)\n- **Unique Selling Point:** (Уникальное торговое предложение)"
*   **Применение:** Декомпозиция сложных задач, гейм-дизайн, продуктовая разработка.

#### 10. Паттерн: "Рефлексивный критик" (Reflective Critic)
*   **Философия:** Двухэтапный промпт. Сначала модель генерирует ответ, потом сама же его критикует и улучшает. Полностью соответствует идеям Andrew Ng.
*   **Компоненты:**
    *   **Шаг 1 (Генератор):** "Напиши короткое эссе о влиянии соцсетей на общество."
    *   **Шаг 2 (Критик - может быть в том же промпте):** "А теперь выступи в роли предвзятого критика. Найди в написанном эссе три слабых аргумента и предложи, как их усилить."
*   **Применение:** Улучшение качества текста, кода, планов. Повышение надежности и глубины ответов.

### **Группа 1: Формальные / "Контрактные" Фреймворки**

(Подход: промпт как строгий технический или юридический документ)

1.  **The Legal Brief (Юридическая Записка):** Структура `[Факты] -> [Применимые Законы/Правила] -> [Анализ] -> [Заключение]`. Идеально для задач, требующих строгого следования правилам.

2.  **The API Contract (Контракт API):** Структура `[Endpoint/Function Name] -> [Description] -> [Parameters (Name, Type, Required)] -> [Return Value Schema]`. Для задач, где LLM должен сгенерировать что-то строго структурированное.

3.  **The Scientific Paper (IMRaD):** Структура `[Introduction/Goal] -> [Methods/Process] -> [Results/Output Spec] -> [Discussion/Reflection]`. Отлично подходит для исследовательских и аналитических задач.

### **Группа 2: Ролевые / "Персонажные" Фреймворки**

(Подход: промпт как сценарий или инструкция для актера)

4.  **The Actor's Method (Система Станиславского):** Структура `[Роль] -> [Сверхзадача] -> [Предлагаемые Обстоятельства (Контекст)] -> [Действие]`. Глубокое погружение модели в роль для генерации очень стилизованных ответов.

5.  **The Expert Panel (Экспертный Совет):** Моделируется диалог нескольких ролей. `[Модератор: Цель] -> [Эксперт A: Взгляд с одной стороны] -> [Эксперт B: Взгляд с другой стороны] -> [Аналитик: Синтез и вывод]`. Для генерации сбалансированных, многогранных ответов.

6.  **The Press Conference (Пресс-конференция):** Структура `[Главный Спикер: Основной тезис] -> [Журналисты: Набор острых вопросов, на которые нужно ответить] -> [Спикер: Финальное заявление]`. Заставляет модель предвидеть и отвечать на потенциальную критику.

### **Группа 3: Процессные / "Алгоритмические" Фреймворки**

(Подход: промпт как алгоритм или описание процесса)

7.  **The Pseudocode Spec (Спецификация в Псевдокоде):** `FUNCTION SolveProblem(input):\n // Step 1: Analyze input...\n // Step 2: Generate draft...\n // Step 3: Verify constraints...\n RETURN result`. Для задач, требующих строгой последовательности логических шагов.

8.  **The User Story (BDD/Gherkin):** `FEATURE: [Название фичи]\n SCENARIO: [Название сценария]\n GIVEN [Исходный контекст]\n WHEN [Происходит действие]\n THEN [Ожидаемый результат]`. Классика для описания поведения.

9.  **The Assembly Line (Сборочный Конвейер):** `[Станция 1: Прием Сырья (Input)] -> [Станция 2: Первичная Обработка (Generate)] -> [Станция 3: Контроль Качества (Critique)] -> [Станция 4: Упаковка (Format Output)]`. Линейный, пошаговый процесс.

### **Группа 4: Минималистичные / "Ограничительные" Фреймворки**

(Подход: минимум инструкций, максимум свободы внутри жестких рамок)

10. **The Zen Koan (Дзенский Коан):** Только `[Цель]` и один `[Ключевой парадокс/вопрос]`. Дает модели максимальную свободу для "творческого" решения внутри очень узкого концептуального коридора.

11. **The Guardrail Protocol (Протокол Ограждений):** Только `[Цель]` и `[Запреты (DO NOT...)]`. Вместо того чтобы говорить, что делать, мы говорим, чего НЕ делать. Полезно для избегания типичных ошибок.

12. **The Sandbox (Песочница):** Только `[Начальное состояние]` и `[Разрешенные действия/инструменты]`. Модель сама решает, как комбинировать разрешенные действия для достижения неявной цели.

### **Группа 5: Данно-центричные / "Примерные" Фреймворки**

(Подход: данные и примеры важнее инструкций)

13. **The Data Frame (Таблица Данных):** Промпт состоит преимущественно из структурированных данных (Markdown таблица, CSV), с минимальной инструкцией "Проанализируй и сделай вывод".

14. **The Example-Driven Template (Шаблон на Примерах):** Классический few-shot, но формализованный. `TEMPLATE:\n Input: {{input}}\n Output: {{output}}\n EXAMPLES:\n [Пример 1]\n [Пример 2]\n EXECUTE:\n Input: [Новый Вход]\n Output:`.

15. **The Contrastive Pair (Контрастная Пара):** `[Цель]\n [Хороший Пример (Good Example)]\n [Плохой Пример (Bad Example) и почему он плох]\n [Твоя задача]`. Обучение на анти-примерах, один из самых мощных методов.


#### Группа 1: Аналогии из Инженерных и Гуманитарных Дисциплин

Эти фреймворки заимствуют проверенные временем структуры из других областей.

1.  **The User Story (`AS A / I WANT / SO THAT`)**
    *   **Структура:** `AS A [Роль/Персона], I WANT [Действие/Цель], SO THAT [Бизнес-ценность]`.
    *   **Применение:** Идеально для постановки бизнес-ориентированных задач. Заставляет сразу думать о конечной ценности.

2.  **The Gherkin (`GIVEN / WHEN / THEN`)**
    *   **Структура:** `GIVEN [Исходное состояние/Контекст], WHEN [Событие/Действие], THEN [Ожидаемый результат]`.
    *   **Применение:** Для задач, где важна логика "если-то". Отлично подходит для написания правил валидации, классификации или триггеров.

3.  **The Legal Brief (`IRAC`)**
    *   **Структура:** `ISSUE [Проблема], RULE [Применимое правило/Принцип], APPLICATION [Применение правила к фактам], CONCLUSION [Вывод]`.
    *   **Применение:** Для сложных аналитических задач, требующих логического вывода на основе набора правил (например, наш фреймворк "Линз").

4.  **The 5 Ws (`WHO / WHAT / WHEN / WHERE / WHY`)**
    *   **Структура:** Пять классических журналистских вопросов для сбора полной картины.
    *   **Применение:** Отлично подходит для генерации саммари, отчетов или первоначального сбора требований по задаче.

5.  **The Scientific Method (`HYPOTHESIS / EXPERIMENT / RESULTS / CONCLUSION`)**
    *   **Структура:** Формализация исследовательского процесса.
    *   **Применение:** Для агентов, которые должны анализировать данные, находить паттерны и делать выводы.

#### Группа 2: Структурно-Компонентные Фреймворки

Эти фреймворки фокусируются на явном определении всех необходимых компонентов для выполнения задачи.

6.  **The R.O.P.E.S.**
    *   **Структура:** `ROLE / OBJECTIVE / PROCESS / EXEMPLARS / SPECIFICATION (Output)`.
    *   **Применение:** Очень полный и строгий фреймворк, похожий на то, что мы обсуждали. Хорош для сложных, критически важных задач.

7.  **The C.O.S.T.A.R.**
    *   **Структура:** `CONTEXT / OBJECTIVE / STYLE / TONE / AUDIENCE / RESPONSE (Format)`.
    *   **Применение:** Сильный акцент на "мягких" аспектах генерации контента. Идеален для Storyteller-агентов.

8.  **The S.P.E.C.**
    *   **Структура:** `SITUATION [Описание текущего положения], PURPOSE [Цель операции], EXPECTATIONS [Детальные ожидания от результата], CRITERIA [Критерии оценки успеха]`.
    *   **Применение:** Хорош для задач, где важна последующая автоматическая или ручная оценка качества результата.

9.  **The I.P.O. (`INPUT / PROCESSING / OUTPUT`)**
    *   **Структура:** Классическая модель из информатики. `INPUT [Входные данные], PROCESSING [Шаги обработки], OUTPUT [Спецификация вывода]`.
    *   **Применение:** Универсальный и простой фреймворк для большинства задач по трансформации данных.

#### Группа 3: Целе-ориентированные и Копирайтерские Фреймворки

Эти фреймворки заточены под конкретный тип результата: убеждение, создание, анализ.

10. **The P.A.S. (`PROBLEM / AGITATE / SOLVE`)**
    *   **Структура:** `PROBLEM [Опиши проблему аудитории], AGITATE [Усугуби эту проблему, покажи ее последствия], SOLVE [Предложи решение]`.
    *   **Применение:** Классика копирайтинга. Идеально для генерации продающих текстов, маркетинговых материалов.

11. **The A.I.D.A. (`ATTENTION / INTEREST / DESIRE / ACTION`)**
    *   **Структура:** Еще одна классическая маркетинговая воронка.
    *   **Применение:** Для генерации контента, который должен провести пользователя по определенному пути.

12. **The Hegelian Dialectic (`THESIS / ANTITHESIS / SYNTHESIS`)**
    *   **Структура:** `THESIS [Изложи исходное утверждение], ANTITHESIS [Сформулируй противоположное утверждение], SYNTHESIS [Создай новый вывод, объединяющий или разрешающий противоречие]`.
    *   **Применение:** Для глубокого анализа сложных тем, генерации разносторонних аргументов или создания сбалансированного обзора.

#### Группа 4: Минималистичные и Специализированные Фреймворки

Для простых или очень специфичных задач.

13. **The Command (`VERB / NOUN / CONSTRAINTS`)**
    *   **Структура:** `[Действие] [Объект] [Ограничения]`.
    *   **Применение:** Для простых, унитарных задач. `GENERATE list of 10 ideas FOR blog_post ABOUT 'AI agents'`.

14. **The Transformer (`INPUT / FORMAT_FROM / FORMAT_TO`)**
    *   **Структура:** `INPUT [Данные], FORMAT_FROM [Исходный формат], FORMAT_TO [Целевой формат]`.
    *   **Применение:** Специализированный фреймворк для задач по преобразованию форматов (JSON в YAML, Markdown в HTML и т.д.).

15. **The Reflector (`INPUT / CRITERIA / FEEDBACK`)**
    *   **Структура:** `INPUT [Черновик для анализа], CRITERIA [Список критериев для оценки], FEEDBACK [Формат для вывода критики]`.
    *   **Применение:** Специализированный фреймворк для реализации паттерна "Рефлексия", о котором говорил Andrew Ng.
